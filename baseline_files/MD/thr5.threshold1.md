## Functional Specification: **`thr5.threshold1`**

### Signal Name

`thr5.threshold1`

### Signal Type

Typically:

```vhdl
thr5.threshold1 : integer
```

or

```vhdl
thr5.threshold1 : unsigned
```

depending on implementation.

### Module Context

Used inside:

* `rgb_kmeans_cluster_engine`
* threshold evaluation stage
* minimum-distance comparison logic

`thr5` generally represents a **threshold computation structure**, and `threshold1` is one of the distance results generated for centroid comparison.

---

# 1. Functional Purpose

`thr5.threshold1` represents the **computed distance between the current input pixel and a specific centroid**.

This signal is one candidate value used in the clustering decision.

It participates in determining:

```text
minimum pixel-to-centroid distance
```

---

# 2. Algorithm Context

In K-means clustering, the engine evaluates:

```
distance(pixel, centroid_i)
```

for multiple centroids.

Each computed distance is stored in a threshold signal.

Example:

| Centroid  | Distance Signal |
| --------- | --------------- |
| centroid0 | `threshold0`    |
| centroid1 | `threshold1`    |
| centroid2 | `threshold2`    |
| centroid3 | `threshold3`    |
| centroid4 | `threshold4`    |

Thus:

```
thr5.threshold1
```

represents the **distance to centroid 1**.

---

# 3. Mathematical Definition

Let the input pixel be:

```
P = (R, G, B)
```

and centroid 1 be:

```
C1 = (R1, G1, B1)
```

Then:

```
thr5.threshold1 =
    |R − R1|
  + |G − G1|
  + |B − B1|
```

This is the **Manhattan distance** in RGB space.

---

# 4. Functional Role in Clustering

`thr5.threshold1` is one of several candidate distances.

All threshold signals participate in the minimum-distance selection stage.

Example candidate set:

```
threshold0
threshold1
threshold2
threshold3
threshold4
```

The clustering engine selects:

```
min_distance = min(threshold0..threshold4)
```

---

# 5. Example Operation

Assume input pixel:

```
P = (100, 120, 140)
```

Centroid 1:

```
C1 = (110, 118, 130)
```

Distance:

```
|100−110| = 10
|120−118| = 2
|140−130| = 10
```

Therefore:

```
thr5.threshold1 = 22
```

---

# 6. Relationship to Other Signals

| Signal        | Relationship                         |
| ------------- | ------------------------------------ |
| `threshold0`  | distance to centroid 0               |
| `threshold1`  | distance to centroid 1               |
| `threshold2`  | distance to centroid 2               |
| `threshold3`  | distance to centroid 3               |
| `threshold4`  | distance to centroid 4               |
| `int_min_val` | smallest of all thresholds           |
| `int_min_id`  | index of threshold producing minimum |

---

# 7. Hardware Computation

`thr5.threshold1` is generated by:

1. subtracting RGB channels
2. computing absolute values
3. summing the results

Conceptual hardware:

```
|R - R1| → diff_r
|G - G1| → diff_g
|B - B1| → diff_b

threshold1 = diff_r + diff_g + diff_b
```

---

# 8. RTL Representation

Typical synthesizable behavior:

```vhdl
diff_r <= abs(pixel_r - centroid1_r);
diff_g <= abs(pixel_g - centroid1_g);
diff_b <= abs(pixel_b - centroid1_b);

thr5.threshold1 <= diff_r + diff_g + diff_b;
```

---

# 9. Data Range

For 8-bit RGB:

```
0 ≤ |R − R1| ≤ 255
0 ≤ |G − G1| ≤ 255
0 ≤ |B − B1| ≤ 255
```

Maximum threshold:

```
255 + 255 + 255 = 765
```

Therefore:

```
0 ≤ threshold1 ≤ 765
```

---

# 10. Timing Behavior

The threshold is typically generated **within one pipeline stage**.

Input timing:

```
pixel_in_rgb @ t
centroid1_rgb @ t
```

Output timing:

```
thr5.threshold1 @ t + 1
```

depending on register placement.

---

# 11. Pipeline Role

`thr5.threshold1` participates in the **distance comparison pipeline**.

Typical pipeline flow:

```
Pixel Input
      │
Distance Engine
      │
threshold0..threshold4
      │
Minimum Distance Selector
      │
int_min_val
int_min_id
```

Thus, `threshold1` is one element of the candidate distance set.

---

# 12. Reset Behavior

During reset:

```
thr5.threshold1 = 0
```

or is undefined until valid input pixels arrive.

Since thresholds are recomputed for every pixel, reset values do not affect clustering results.

---

# 13. Design Intent

The purpose of `thr5.threshold1` is to provide the distance metric between the current pixel and centroid 1 so the clustering engine can determine whether centroid 1 is the closest centroid.

It is part of the **parallel distance computation stage** of the clustering architecture.

---

# 14. Advantages

* simple arithmetic
* efficient FPGA implementation
* avoids multipliers
* deterministic timing
* scalable across many centroids

---

# 15. Limitations

* Manhattan distance is an approximation of Euclidean distance
* large centroid counts increase comparator complexity
* thresholds must remain synchronized with centroid index tracking

---

centroid1_distance

# 17. Document-Ready Functional Description

`thr5.threshold1` represents the Manhattan distance between the current input pixel and centroid 1. It is generated by summing the absolute differences between the pixel’s red, green, and blue components and the corresponding centroid components. This signal is one of the candidate distances evaluated by the minimum-distance selection logic to determine the closest centroid during the clustering process.

---

# 18. Short Engineering Definition

`thr5.threshold1` is the computed RGB Manhattan distance between the input pixel and centroid 1.

---

# 19. Summary

`thr5.threshold1`:

* stores distance between pixel and centroid 1
* is part of the candidate distance set
* feeds the minimum-distance decision logic
* participates in determining the winning centroid
* contributes to cluster assignment
